# План работ по АСУ управления ТРК по протоколу Tatsuno

Документ агрегирует выводы из `docs/protocol_analysis.md`, `docs/refactoring_summary.md`, `docs/hang_diagnosis.md`, `docs/IMPLEMENTATION_NOTES_chatgpt.md`, `docs/tatsuno.txt` и `docs/readme.txt` и формулирует дорожную карту дальнейших работ до уровня полноценной АСУ для управления ТРК по протоколу Tatsuno.

## 1. Цель и границы системы

- Обеспечить **надёжное управление ТРК** по протоколу Tatsuno:
  - опрос состояния постов и пистолетов;
  - авторизация отпуска по сумме/объёму;
  - установка цен;
  - запрос итогов (totals);
  - обработка ошибок и аварийных состояний.
- Построить **двухслойную архитектуру**:
  - `Tatsuno.Native` (C++) — протокольное ядро, FSM, transaction scheduler, модель постов/пистолетов;
  - `Tatsuno.WpfApp` (C# WPF) — только UI/операторский интерфейс и интеграция с C++ через C API.
- Обеспечить возможность **масштабирования**:
  - несколько постов/адресов в одной системе;
  - несколько пистолетов на пост;
  - конфигурируемые decimals/форматы для разных колонок.

## 2. Текущее состояние проекта (по документам)

- **Уже есть**:
  - Разделение на проекты: `Tatsuno.Model`, `Tatsuno.Protocol`, `Tatsuno.Transport.Serial`, `Tatsuno.WpfApp`, заготовка `Tatsuno.Native`.
  - Работающий прототип опроса по COM: кадры вида `<EOT>@Q<ENQ>`, `<EOT>@A<ENQ>`, обработка `STX ... ETX + BCC`, ACK (`<DLE>1`), базовая логика polling/select.
  - Проверенная реализация **BCC** (XOR payload+ETX), корректно сходится с эталонным логом `docs/tatsuno.txt`.
  - Переработанная модель данных в `Tatsuno.Model` (см. `refactoring_summary.md`):
    - `PumpOperationalState`, `LinkLayerState`, `TransactionType`, `PumpCondition`;
    - `NozzleState`, `PumpAddressState`, обновлённый `PostStatus` с raw-значениями и decimals.
- **Сделан анализ проблем** (`protocol_analysis.md`, `readme.txt`):
  - неправильное масштабирование цены и суммы;
  - бедная модель данных в старом `PostStatus`;
  - отсутствие поддержки всех типов Q6xx кадров (Q612, Q613, Q614, Q651 и др.);
  - отсутствие FSM для link-layer и постов/пистолетов;
  - проблемный poll loop в `MainViewModel` (бесконечный опрос без координации с select/transactions);
  - баги логирования и возможные зависания WPF (см. `hang_diagnosis.md`).
- **Рефакторинг Фазы 1–2** уже выполнен (`refactoring_summary.md`):
  - очищен мусорный код, созданы новые модели, исправлена концепция хранения raw+decimals.

## 3. Целевая функциональность АСУ

К моменту окончания работ система должна обеспечивать:

1. **Полную поддержку протокола Tatsuno** (в рамках нужного вам подмножества):
   - корректный разбор кадров Q610/611/612/613/614/651 и др.;
   - обработка totals, error/status/switch кадров;
   - поддержка всех необходимых action-команд (authorize, cancel, totals, set price, lock/release и т.п.).
2. **Надёжную FSM протокольного ядра**:
   - link-layer FSM: poll/select, retries, timeouts, fault-состояния;
   - FSM поста/пистолета: Idle → Lifted → Authorized → Fuelling → Finished → Totals → Completed/Error.
3. **Транзакционный планировщик**:
   - очередь команд;
   - гарантия последовательной отправки команд на фоне polling;
   - управление жизненным циклом транзакций.
4. **Современный операторский UI (WPF)**:
   - вертикальный список постов по адресу;
   - отдельные цены по каждому пистолету;
   - выбор активного поста (курсором и/или автоматически по подъёму пистолета);
   - ввод пресетов по сумме/объёму и запуск/отмена транзакций;
   - индикация хода отпуска, ошибок, блокировок.
5. **Инженерные требования**:
   - стабильность (отсутствие зависаний UI, корректная работа таймеров и потоков);
   - логирование протокольного трафика и событий;
   - возможность диагностики и удалённой поддержки;
   - предсказуемое поведение при обрыве связи/питания.

## 4. Направления работ и этапы

Ниже — предложенная дорожная карта, основанная на уже описанных в документах фазах (3–7) и расширенная до уровня полноценной АСУ.

### Этап 0. Наведение порядка и фиксация базовой модели (P0)

- Завершить очистку мусорного кода:
  - удалить все оставшиеся `Class1.cs` и неиспользуемые заглушки;
  - убедиться, что `Tatsuno.Model` отражает актуальную модель (`PumpAddressState`, `NozzleState`, обновлённый `PostStatus`).
- Зафиксировать формат **snapshot-модели**, которую будет отдавать C++ ядро:
  - структура `PumpAddressState` с полями адреса, состояний, raw-значений и decimals;
  - массив `NozzleState` внутри каждого адреса;
  - статусы FSM (comm/transaction).
- Уточнить и задокументировать **конфигурацию decimals** для вашей конкретной ТРК:
  - сколько знаков у пути, цены, суммы;
  - как это отображается на реальном табло.

### Этап 1. Протокольный парсер и модель данных (P0)

Опираясь на `protocol_analysis.md` и `refactoring_summary.md`:

- Переписать парсер Q6xx кадров в `Tatsuno.Protocol`:
  - извлекать **все числовые поля как raw integers**;
  - извлекать и/или конфигурировать **decimals** по каждому полю;
  - корректно разбирать:
    - `Q610/611` (статус/индикация);
    - `Q612/613/614` (динамический статус во время отпуска — по документации протокола);
    - `Q651` и другие totals-кадры;
    - кадры ошибок/аварий.
- Заполнить модель:
  - `PumpAddressState` — последнее состояние по адресу;
  - `NozzleState` — состояние каждого пистолета;
  - связи `Address → Nozzles[] → raw/decimals`.
- Перенести **форматирование** (литры, суммы, цены) полностью на уровень UI:
  - в модели хранить только raw+decimals;
  - для WPF — предоставить готовые вычисляемые свойства или функции форматирования.

### Этап 2. Link-layer FSM и TX-очередь (P0)

Опираясь на рекомендации из `protocol_analysis.md` и `readme.txt`:

- Спроектировать и реализовать **link-layer FSM** (изначально можно в C# для отладки, затем перенести в C++):
  - состояния: Idle, SendPoll, WaitPollReply, PollFrameReceived, SendAck1, SendSelectEnq, WaitDle0, SendCommandText, WaitDle1OrNak, ResumePolling, Retry, Fault;
  - явные переходы по событиям: RX `EOT/DLE0/DLE1/NAK/STX...`, таймауты, ошибки порта.
- Ввести **единый TX scheduler/queue**:
  - один поток/компонент, который имеет эксклюзивный доступ к `SerialPort` (или к каналу RX/TX, если COM останется в C#);
  - все операции записи (poll, select, commands) идут только через очередь, с учётом состояния FSM;
  - гарантируется, что во время select/transaction poll либо остановлен, либо строго управляется FSM.
- Настроить **timeouts и retries**:
  - inter-byte timeout / frame timeout в декодере;
  - таймауты ожидания ответов на команды (Dle0/Dle1/Nak/Eot);
  - стратегия повторов и перехода в Fault.

### Этап 3. FSM постов/пистолетов и транзакционный планировщик (P0/P1)

- Реализовать **FSM поста/пистолета**:
  - состояния: Unknown, Idle, NozzleLifted, PricePending, PresetPending, Authorized, Fuelling, Finished, TotalsRequested, Completed, Error;
  - переходы по событиям из кадров Q6xx и action-команд.
- Ввести **transaction scheduler**:
  - очередь транзакций (по адресу и пистолету);
  - шаги транзакции:
    - выбор поста (select);
    - установка цены (по необходимости);
    - установка пресета (сумма/объём);
    - авторизация;
    - мониторинг хода отпуска;
    - запрос totals;
    - закрытие транзакции.
  - правила очередности при нескольких одновременных запросах.
- Определить и реализовать **интерфейс событий** для UI/АСУ:
  - события о смене состояния поста/пистолета;
  - события о старте/завершении транзакций;
  - события об ошибках/авариях.

### Этап 4. Вынесение ядра в C++ (`Tatsuno.Native`) и C API (P0)

- Сформулировать минимально необходимый **C API** (см. `readme.txt`):
  - создание/уничтожение контроллера: `tm_create_controller(...)`, `tm_destroy_controller(...)`;
  - приём байтов RX: `tm_feed_rx_bytes(...)` (если COM оставляем в C#) или прямое управление портом из C++;
  - тиковая функция: `tm_tick(...)` для обработки таймеров и FSM;
  - операции управления:
    - `tm_select_post(address)`;
    - `tm_set_nozzle_price(address, nozzle, price_raw, decimals)`;
    - `tm_start_amount(address, nozzle, amount_raw)`;
    - `tm_start_volume(address, nozzle, volume_raw)`;
    - `tm_cancel_transaction(address)`;
    - `tm_request_totals(address)`;
  - получение snapshot: `tm_get_snapshot(...)` (структура с `PumpAddressState[]` и `NozzleState[]`).
- Перенести в C++:
  - парсер кадров;
  - link-layer FSM;
  - FSM постов/пистолетов;
  - transaction scheduler;
  - модель состояний и конфигураций.
- Настроить **P/Invoke** в WPF-приложении для работы с C API:
  - безопасные обёртки для структур и указателей;
  - периодическое опросное обновление snapshot с маппингом в ViewModels.

### Этап 5. Взаимодействие с COM и транспортный слой (P1)

- Определиться, где физически живёт **COM-транспорт**:
  - вариант A: COM-порт управляется C#, а байты передаются в C++ через `tm_feed_rx_bytes` / `tm_get_tx_bytes`;
  - вариант B: COM переносится в C++ и управляется внутри `Tatsuno.Native`.
- В обоих случаях:
  - обеспечить однократный контроль за TX/RX;
  - реализовать отчёт об ошибках порта и reconnect;
  - согласовать timeouts и буферизацию с FSM.

### Этап 6. WPF UI: операторский интерфейс (P1)

Опираясь на описания в `IMPLEMENTATION_NOTES_chatgpt.md` и `readme.txt`:

- Переработать структуру UI:
  - вертикальный список постов по адресу вместо “строки по индексу”;
  - в карточке поста:
    - адрес поста;
    - список пистолетов с ценой каждого;
    - текущие показания (объём, сумма, цена) в формате, совпадающем с колонкой;
    - индикация состояния (цвет/иконки для Idle, Lifted, Authorized, Fuelling, Error и др.).
- Ввести понятие **активного поста**:
  - выбор курсором/кликом;
  - автоматический выбор при подъёме пистолета (lifted имеет приоритет);
  - визуальное выделение активного поста.
- Реализовать **панель транзакций**:
  - выбор типа пресета (по сумме/по объёму/без пресета);
  - поле ввода значения;
  - кнопки `Authorize`, `Cancel`, `Totals`, `Lock`, `Release` и др. в привязке к активному посту/пистолету;
  - отображение статуса текущей/последней транзакции.
- Обновить **логирование в UI**:
  - структурированный лог (время, тип события, адрес, пистолет, текст);
  - фильтры по типам событий (RX/TX/SYS/ERR).

### Этап 7. Логирование, диагностика и стабильность (P1)

- Исправить текущие проблемы логирования (см. `hang_diagnosis.md`, `protocol_analysis.md`, `readme.txt`):
  - убрать некорректный вызов `CloseFileLog()` из `StartPolling()` (или его аналог в новой архитектуре);
  - убедиться, что file-log стабильно работает на всём жизненном цикле сессии.
- Перепроверить использование `Dispatcher.Invoke()` / `BeginInvoke()`:
  - заменить блокирующие вызовы на неблокирующие там, где это безопасно;
  - устранить потенциальные deadlock-и и зависания UI.
- Перепроверить XAML-binding-и:
  - для read-only свойств использовать `Mode=OneWay`, чтобы избежать циклических обновлений.
- Добавить **диагностические счётчики и health-check-и**:
  - количество полученных/отброшенных кадров;
  - количество ошибок BCC/таймаутов;
  - текущее состояние FSM (для отладки).

### Этап 8. Тестирование на реальном оборудовании и ввод в эксплуатацию (P0/P1)

- Разработать **план тестирования**:
  - сценарии нормальной работы (один пост, несколько постов, несколько пистолетов);
  - сценарии ошибок (нет связи, обрыв в середине отпуска, NAK от ТРК, аварийная остановка);
  - сценарии восстановления (reconnect, повторная авторизация, повторные totals).
- Организовать **трассировку с реального железа**:
  - сравнивать `tatsuno.txt` и новые логи;
  - подтверждать корректность парсинга и FSM по логам.
- Подготовить **эксплуатационную документацию**:
  - для операторов (как работать с UI);
  - для инженеров (как настраивать decimals, адреса постов, параметры COM, сценарии диагностики).

## 5. Приоритизация (P0/P1/P2)

- **P0 (критический минимум для боеспособного ядра):**
  - Этап 0 — фиксация модели и конфигурации decimals.
  - Этап 1 — полноценный парсер и модель данных.
  - Этап 2 — link-layer FSM и TX-очередь.
  - Ключевые части Этапа 3 — FSM постов/пистолетов и базовый transaction scheduler.
  - Этап 4 — перенос ядра в `Tatsuno.Native` с C API.
  - Базовые тесты с реальным оборудованием (часть Этапа 8).

- **P1 (операторский комфорт и удобство сопровождения):**
  - Завершение Этапа 3 (расширенный планировщик).
  - Этап 5 — транспортный слой и обработка ошибок порта.
  - Этап 6 — новый WPF UI под оператора.
  - Этап 7 — расширенное логирование и диагностика.
  - Полный план тестирования и документация (Этап 8).

- **P2 (дальнейшие улучшения и расширения):**
  - Поддержка дополнительных режимов и команд протокола, не критичных для текущей эксплуатации.
  - Интеграция с внешними системами АЗС (учёт, касса, отчётность).
  - Расширение UI (история транзакций, отчёты, дистанционный мониторинг).
